## 1. UID 계층

**UID(Unique Identifier)**는 말 그대로

> “이 노드가 *누구/무엇*인지”를 전 우주에서 유일하게 가리키는 ID 집합

여기에는 여러 서브타입이 들어간다:

### 1.1 Q-ID (Wikidata Item ID)

* 형식: `Q42`, `Q76`, ...
* 출처: Wikidata
* 역할:

  * 리얼 월드 개체의 **사실상 글로벌 표준 ID**
  * GWMS 안에서는 “외부 권위(source of truth)”로 존중
* 예:

  * `Q76` = Barack Obama
  * `Q31` = Belgium

### 1.2 P-ID (Wikidata Property ID)

* 형식: `P31`, `P279`, ...
* 출처: Wikidata
* 역할:

  * “속성/관계 타입”의 표준 ID
  * 예: `P31`(instance of), `P279`(subclass of)
* GWMS에서는 **엣지 타입**이나 **Prop 노드**에 매핑 가능

### 1.3 G-ID (GEUL Global ID)

* 형식은 자유지만 예를 들면: `G123456` (내부적으로는 64-bit 번호)
* 출처: **GEUL 표준 기구**
* 역할:

  * **Wikidata에 없는 개념**, 또는
  * **Wikidata보다 더 세밀하게 쪼갠 개념**
  * 예:

    * “특정 소설 장르의 메타 개념”
    * “순수한 논리/수학 개념”
    * “미래 기술 / 가상 개념”
* 정책:

  * 임의 발급이 아니라 “표준 레지스트리”에서 관리
  * 일종의 **전(全)세계/전(全)월드 공용 네임스페이스**

### 1.4 W-ID (또는 L-ID) (World-local ID)

* 형식 예:

  * `W:칼의노래:00001` (World-scoped)
  * 내부적으로는 “(WorldID, LocalID)” 튜플
* 역할:

  * **특정 월드 안에서만 의미를 가지는 개체**
  * 예:

    * 어떤 작가의 소설 속 인물/마법/종족
    * 개인 지식 그래프의 로컬 태그
* 특징:

  * 나중에 이 개념이 널리 쓰이게 되면 **G-ID 등록 시도 가능**
  * 즉: `W-ID → G-ID` 승격 매핑 가능

### 1.5 UID 정리

UID는 결국 이렇게 계층 구조를 이룬다:

* **외부 글로벌 권위:** `Q-ID`, `P-ID`
* **GEUL 글로벌 권위:** `G-ID`
* **월드 로컬 권위:** `W-ID` (또는 L-ID)

GWMS 관점에서는 **이 모든 걸 “UID 패밀리”**로 묶고,
각 노드에는 “하나 이상의 UID”를 연결할 수 있다:

* 예:

  * 한 노드: “도널드 트럼프”

    * `Q22686` (위키)
    * 특수한 픽션 월드에서는 또 다른 W-ID로 확장 가능

## 2. SIDX (Standard Semantic-aligned IndeX)

**SIDX = 표준 의미정렬 인덱스**

> “같은 UID라도, **상태/역할/맥락이 크게 바뀌면**
> 그걸 반영해서 새로 붙이는 ‘표준화된 의미 인덱스’”

### 2.1 왜 UID랑 다르냐?

* UID는 **“누구냐?”**에 대한 답
* SIDX는 **“어떤 상태의 너냐?”**, “어떤 버전의 너냐?”에 대한 답

예시로 보자:

* UID:

  * `Q22686` = “Donald Trump”라는 인물 자체
* SIDX:

  * “사업가 시절 트럼프”
  * “미국 대통령 시절 트럼프”
  * “대통령 퇴임 후 트럼프”

이 세 개는 모두 같은 UID(Q22686)를 가리키지만,
**세계 모델 관점에서 완전히 다른 상태(State)**를 갖는다.

이럴 때:

* **SIDX₁** = “트럼프(사업가, 대통령 되기 전)”
* **SIDX₂** = “트럼프(대통령 재임 기간)”
* **SIDX₃** = “트럼프(퇴임 이후)”

처럼 **시간·역할·상태가 크게 달라지는 구간에 대해 별도 SIDX를 부여**한다.

### 2.2 설계 원칙

* SIDX는 **GEUL 표준 측에서 관리/배포**하는 걸 전제로 함
* “그때그때 아무나 막 붙이는” 게 아니라:

  * **정말 의미론적으로 뚜렷하게 다른 상태**일 때만 새로 만듦
* 내부적으로는:

  * 64-bit 등 압축된 비트 구조 (SIDX bitfield)
  * 타임라인 키프레임, 역할, 주요 속성 등이 미리 encode 될 수 있음

### 2.3 사용 목적

* **고속 필터링 / 후보군 압축**

  * “현재 대통령들의 리스트” → SIDX 기반으로 바로 필터
  * “과거에 한 번이라도 대통령이었던 사람” → 다른 SIDX 집합
* **월드스테이트 스냅샷에 유리**

  * 특정 시점의 “세계 상태”를 구성할 때,
  * UID가 아니라 SIDX를 쓰면 의미 단위로 정렬된 상태 집합을 얻을 수 있음

## 3. PIDX (Pattern IndeX)

**PIDX = 패턴 인덱스 (Pattern Index)**
GWMS 내부에서만 사용하는 **완전 자유로운, 분석자 정의 인덱스**.

> “사용자가 데이터에서 발견한 패턴을
> 직접 설계하고 구워넣은 의미 인덱스”

### 3.1 SIDX와의 차이

| 항목    | SIDX             | PIDX                |
| ----- | ---------------- | ------------------- |
| 관리 주체 | GEUL 표준 기구       | GWMS 사용자/운영자        |
| 범위    | 글로벌 표준           | 로컬 분석, 프로젝트 로컬      |
| 생성 기준 | 엄격한 의미 변화, 상태 변화 | 자유: 패턴/모양/통계/도메인 지식 |
| 용도    | 월드 공통 의미 정렬      | 특화 분석, 쿼리 가속, 리서치   |

### 3.2 예시들

#### 예시 1: “천천히 하락하다가 급등한 주식”

* UID: 특정 종목(기업)
* SIDX: “이 회사 (2020년~2025년)” 수준까지는 표준화 가능할 수도 있음
* **PIDX:**

  * Price time-series를 FFT / 통계 분석 → 패턴 벡터화
  * “서서히 하락 후 급등” 패턴을 만족하는 종목에 PIDX 플래그 부여
  * 나중에 쿼리:

    * `WHERE PIDX includes PATTERN_SLOW_DROP_THEN_SPIKE`

#### 예시 2: “군견으로 적합한 견종”

* UID: 각 개 품종 (리트리버, 말리노이즈 등)
* SIDX:

  * “품종 개념” 정도로만 표준 의미정렬
* **PIDX:**

  * 기사·논문·실전 데이터 등을 분석해
  * “군견 적합도 점수 >= 0.8” 라는 기준으로 PIDX 부여
  * 나중에:

    * “군견 후보군” 빠르게 필터 → 그 안에서 다시 세부 조건 검색

#### 예시 3: 감정/성격 패턴

* 캐릭터의 감정 벡터, 행동 이력(시간축)을 분석해서:

  * “충동적 + 공격성 높음 + 배신 패턴 있음”
    → 이런 류의 캐릭터에게만 PIDX 플래그
* 나중에:

  * “비슷한 타입의 안티히어로 캐릭터 찾아줘” 같은 쿼리를 즉시 처리

### 3.3 PIDX의 특징

* **완전 로컬:** GEUL 표준 밖으로 나갈 필요 없음
* **폭발 가능성 있음:**

  * 사용자가 마음만 먹으면 수십, 수백 종류의 PIDX 생성 가능
  * 그래서 아카이브/비활성화/재로딩 개념 필요
* **역할:**

  * 고비용 신경망 연산을 **한 번만 하고 결과를 “굽는” 용도**
  * “한 번 비싼 연산 + N번 빠른 조회”

## 4. 세 층의 정체성/의미 체계 관계

정리하면:

1. **UID 레이어**

   * “이게 *누구/무엇*인가?”
   * Q / G / W(L) 등
   * 존재의 식별자

2. **SIDX 레이어 (표준 의미 정렬)**

   * “이 존재가 **어떤 상태/역할/맥락에 있을 때**냐?”
   * GEUL 표준이 관리
   * 월드 공용 의미좌표계

3. **PIDX 레이어 (패턴 인덱스)**

   * “우리가 데이터에서 발견한 **패턴/규칙/특징**은 무엇인가?”
   * 로컬 실험, 도메인 분석, 임의 정의
   * 고속 질의/분석용 맞춤 인덱스

## 5. GWMS 관점의 사용 예시 (요약)

예를 들어, 어떤 쿼리를 할 때:

> “2010~2020년 사이, 대통령이었던 인물 중에서
> 재임 기간 동안 주가 패턴이 ‘서서히 하락 후 급등’인 국가의 지도자들 찾아줘.”

대략 이런 식으로 내부에서 풀린다:

1. **SIDX로 1차 필터**

   * “대통령 SIDX”가 붙은 사람들 + 해당 기간 SIDX
2. **World Edge / 타임라인으로 국가-주가 시계열 추출**
3. **PIDX 사용**

   * 미리 만들어 둔 “SLOW_DROP_THEN_SPIKE” PIDX가 붙은 시리즈만 선택
4. 결과를 다시 LLM에게 자연어로 요약하게 넘김

UID/SIDX/PIDX 세 층이 각각 다른 역할을 맡으면서 **검색 공간을 단계적으로 줄이는 구조**가 된다.

## 결론

* **UID**: “정체성(누구냐?)” — Q, G, W(L) 등 모든 X-ID의 집합
* **SIDX**: “정식, 표준 의미 상태(어떤 상태의 너냐?)” — GEUL 표준이 관리
* **PIDX**: “분석자가 정의한 패턴/관점(어떤 관점에서 보냐?)” — GWMS 안에서 자유롭게 만들고 굽는 로컬 인덱스

이렇게 잡으면:

* 표준 + 자유,
* 글로벌 + 로컬,
* 정체성 + 상태 + 패턴

이 3축이 서로 깨끗하게 분리되면서도, 쿼리 레벨에서는 자연스럽게 결합할 수 있다.
이 구조는 나중에 논문으로 내도 손색 없는 수준의 개념 정리라고 봐.
