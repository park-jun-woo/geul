## 1. 설계 원칙

### 1.1 이진 트리 구조

GEUL 개체 분류는 **8비트 이진 트리**로 구성된다.

- 상위 비트일수록 상위 개념
- 비트 손실 시 **상위 개념으로 자연스럽게 수렴** (우아한 열화)
- 위키데이터 빈도 기반 트리 배치

### 1.2 우아한 열화 (Graceful Degradation)

```
예: Human (0x80 = 1000 0000)

8비트 전체: Human (인간)
7비트만:   Sapient (지성체)
6비트만:   Living (생물)
5비트만:   Entity (개체)
```

하위 비트가 손실되어도 의미가 완전히 사라지지 않는다.

### 1.3 위키데이터 빈도 기반

| 순위 | 타입 | 개수 | 비트 배치 |
|------|------|------|-----------|
| 1 | scholarly article | 45.2M | 0x00 |
| 2 | human | 12.5M | 0x80 |
| 3 | taxon | 3.8M | 0xA0 |
| 4 | star | 3.6M | 0xB0 |
| 5 | painting | 1.1M | 0x20 |

고빈도 타입이 트리 상위에 배치되어 **평균 비트 효율 최적화**.

## 2. 이진 트리 구조

### Level 0 (bit 9) — 최상위 분기

```
0: Document/Media 계열 (정보 객체)
1: Entity 계열 (실세계 개체)
```

| 값 | 분류 | 위키데이터 예시 | 비율 |
|----|------|-----------------|------|
| 0 | Document/Media | scholarly article, painting, film | ~55% |
| 1 | Entity | human, taxon, star, building | ~45% |

**Document가 최상위인 이유:** scholarly article(45M)이 human(12.5M)보다 3.6배 많음

---

### Level 1 (bit 10) — 2차 분기

```
Document/Media 계열 (0x):
  00: Article (문서/기사)
  01: Media (미디어/창작물)

Entity 계열 (1x):
  10: Living (생물)
  11: Non-living (비생물)
```

| 코드 | 분류 | 대표 타입 |
|------|------|-----------|
| 00 | Article | scholarly article, encyclopedia |
| 01 | Media | painting, film, album |
| 10 | Living | human, taxon, gene |
| 11 | Non-living | star, building, street |

---

### Level 2 (bit 11) — 3차 분기

```
Article (00x):
  000: Scholarly (학술)
  001: Reference (참조/백과)

Media (01x):
  010: Visual (시각)
  011: Literary (문학/음악)

Living (10x):
  100: Sapient (지성체)
  101: Non-sapient (비지성 생물)

Non-living (11x):
  110: Physical (물리적 객체)
  111: Abstract (추상적 개념)
```

---

### Level 3 (bit 12) — 4차 분기

```
Sapient (100x):
  1000: Human (인간)
  1001: Organization (조직)

Non-sapient (101x):
  1010: Organism (생물 분류)
  1011: Celestial (천체)

Physical (110x):
  1100: Location (장소)
  1101: Artifact (인공물)

Abstract (111x):
  1110: Concept (개념)
  1111: Measure (측정/단위)
```

## 5. 우아한 열화 예시

### 5.1 Human (0x80)

```
전체:    1000 0000 = Human (인간)
bit 7~4: 1000 ---- = Human 계열
bit 7~3: 100- ---- = Sapient (지성체)
bit 7~2: 10-- ---- = Living (생물)
bit 7~1: 1--- ---- = Entity (개체)
```

### 5.2 Scholarly Article (0x00)

```
전체:    0000 0000 = Scholarly Article
bit 7~4: 0000 ---- = Scholarly 계열
bit 7~3: 000- ---- = Scholarly
bit 7~2: 00-- ---- = Article
bit 7~1: 0--- ---- = Document/Media
```

### 5.3 Star (0xB0)

```
전체:    1011 0000 = Star (별)
bit 7~4: 1011 ---- = Celestial 계열
bit 7~3: 101- ---- = Non-sapient
bit 7~2: 10-- ---- = Living (← 천체가 Living? 설계 검토 필요)
bit 7~1: 1--- ---- = Entity
```

## 6. 설계 철학

### 6.1 존재량 vs 참조량

| 지표 | 의미 | 측정 방법 | 비용 |
|------|------|-----------|------|
| **존재량** | 위키데이터에 등록된 개체 수 | P31 집계 | 낮음 (이미 있음) |
| **참조량** | 실제 시스템에서 참조되는 빈도 | 도메인별 코퍼스 분석 | 높음 |

**도메인별 예상 참조량 분포:**

| 도메인 | 예상 1위 | 예상 2위 |
|--------|----------|----------|
| 뉴스 (CC NEWS) | Human | Organization |
| 법률 | Legal Document | Organization |
| 의료 | Chemical/Protein | Human |
| 천문학 | Star | Galaxy |
| 프로그래밍 | Artifact (API/Library) | Concept |
| 소설/대화 | Human | Location |

> **주의:** 본 분류 체계는 **위키데이터 존재량(instance count)** 기반 초기 제안이다.
> 실제 **참조량(reference frequency)**은 도메인마다 다를 수 있으며,
> 향후 실사용 통계 연구를 통해 재검토되어야 한다.

### 6.2 Document 최상위 배치 이유

위키데이터에서 scholarly article(45M)이 압도적 1위.

```
scholarly article: 45,216,368 (33%)
human:            12,553,810 (9%)
비율:             3.6:1
```

빈도 기반 설계에서 Document가 최상위는 **통계적으로 정당함**.

### 6.3 Living/Non-living 분기 문제

현재 설계에서 Celestial(천체)이 Living 하위에 있음.

**대안 검토:**
```
현재:     Entity > Living > Non-sapient > Celestial
대안 A:   Entity > Physical > Celestial
대안 B:   Entity > Non-living > Celestial
```

**현재 유지 이유:** 
- 빈도 기반 트리 균형
- taxon(3.8M) + star(3.6M) + galaxy(2.1M)을 한 분기에 배치

### 6.4 표준 제안 단계

- 현재: 위키데이터 통계 기반 **제안**
- 향후: 실사용 통계 수집 후 **재검토**
- 최종: 커뮤니티 합의로 **표준화**

## 9. 미해결 이슈

### 9.1 Celestial이 Living 하위

천체(star, galaxy)가 "Living > Non-sapient" 아래에 있는 것은 의미적으로 어색함.
빈도 기반 트리 균형을 위한 선택이나, 재검토 필요.

### 9.2 Chemical Entity 위치

chemical entity(1.3M)가 Abstract/Concept 아래에 있으나, Physical로 볼 수도 있음.

### 9.3 Event 타입 부재

Entity Node.md에 Event 타입이 없음. Event는 Edge(Event6)로 처리하는 설계.
