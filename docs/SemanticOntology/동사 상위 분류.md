## 1. 설계 원칙

### 1.1 비트 할당 방식

GEUL 동사 분류는 **sub_primitive 단위로 직접 허프만 코딩**을 적용한다.

- Primitive(대분류)는 **개념적 그룹핑**만 담당하며 비트 할당 없음
- Sub_primitive(소분류) 68개에 **빈도 기반 가변 길이 코드** 할당
- 고빈도 동사군일수록 짧은 코드 (4비트 ~ 8비트)

### 1.2 코드 길이 분포

| 코드 길이 | 개수 | 동사 수 합계 |
|-----------|------|-------------|
| 4비트 | 4개 | 6,368 (46.3%) |

## 5. 동사 본문 구조 (16비트)

Verb Edge의 동사 본문 16비트는 다음과 같이 구성된다:

```
┌─────────────────────────┬────────────────────────────┐
│   sub_primitive code    │     트리 내 DFS index      │
│   (4-8비트, 허프만)      │     (8-12비트)             │
└─────────────────────────┴────────────────────────────┘
```

- **sub_primitive code:** 4~8비트 가변 (본 문서의 코드)
- **DFS index:** 해당 sub_primitive 내 개별 동사 식별

### 5.1 DFS index 비트 계산

| sub_primitive 동사 수 | 필요 비트 |
|----------------------|-----------|
| 1~256 | 8비트 |
| 257~512 | 9비트 |
| 513~1024 | 10비트 |
| 1025~2048 | 11비트 |
| 2049~4096 | 12비트 |

### 5.2 예시: CHANGE-TRANSFORM

- sub_primitive code: `0000` (4비트)
- 동사 수: 3,049개 → 12비트 필요
- 총: 4 + 12 = 16비트 ✓

### 5.3 예시: MOVE-FOLLOW

- sub_primitive code: `11000011` (8비트)
- 동사 수: 3개 → 2비트 필요 (최소 8비트 사용)
- 총: 8 + 8 = 16비트 ✓

## 6. 평균 코드 길이 분석

### 6.1 가중 평균 계산

```
평균 = Σ(코드길이 × 동사수) / 총동사수

= (4×6368 + 5×2479 + 6×2321 + 7×1786 + 8×813) / 13767
= (25472 + 12395 + 13926 + 12502 + 6504) / 13767
= 70799 / 13767
≈ 5.14비트
```

### 6.2 고정 할당 대비 절감

| 방식 | 평균 비트 |
|------|-----------|
| 고정 7비트 (68개) | 7.00 |
| 허프만 코딩 | 5.14 |
| **절감** | **1.86비트 (27%)** |

## 7. 설계 철학

### 7.1 허프만 코딩 선택 이유

- CHANGE-TRANSFORM (22.1%)이 압도적 고빈도
- 상위 4개 sub_primitive가 전체의 46.3% 차지
- 고정 비트 할당 시 평균 코드 길이 증가
- 허프만 코딩으로 **평균 비트 수 27% 절감**

### 7.2 Primitive 비트 제거 이유

- 기존: Primitive 3비트 + Sub_primitive 4비트 = 7비트 고정
- 변경: Sub_primitive 직접 코딩 = 4~8비트 가변
- 효과: 고빈도 동사에서 **최대 4비트 절감**

### 7.3 의미적 그룹핑 유지 이유

- Primitive 분류는 **인간 가독성**을 위해 유지
- LLM 학습 시 **의미 클러스터링** 힌트 제공
- 문서화 및 디버깅 용이
