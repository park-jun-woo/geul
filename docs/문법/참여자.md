# 참여자 (Participant)

**버전:** v0.2  
**작성일:** 2026-01-26  
**목적:** 사건 내부의 순수 의미역 정의

---

## 1. 정의

**참여자(Participant)**는 서술(Statement) 내에서 사건에 관여하는 개체의 **의미적 역할**을 명시하는 Edge입니다.

```
Event Node (동사)
    ├─ PARTICIPANT Edge (role=Agent) ──→ Entity Node
    ├─ PARTICIPANT Edge (role=Theme) ──→ Entity Node
    └─ PARTICIPANT Edge (role=Instrument) ──→ Entity Node
```

---

## 2. 설계 원칙

### 2.1 분리 원칙

| 구분 | 소속 | 예시 |
|------|------|------|
| **참여자** | Event 레벨 | Agent, Theme, Recipient |
| **화용 정보** | Context/Claim 레벨 | Speaker, Listener, Evidentiality |

Speaker(화자), Listener(청자), Source(정보출처)는 참여자가 아닌 **동사 한정자** 또는 **Context/Claim**에서 처리합니다.

### 2.2 인코딩

- **4비트** (0x0 ~ 0xF)
- 최대 16개 의미역
- SIMD 비트 연산으로 패턴 매칭 가능

---

## 3. 의미역 목록 (16개)

### 3.1 핵심 참여자 (Core Participants)

| ID | 코드 | 역할 | 정의 | 예시 |
|----|------|------|------|------|
| 0x0 | **AGT** | Agent (행위자) | 의도적으로 행동을 수행하는 주체 | "**철수가** 공을 찼다" |
| 0x1 | **EXP** | Experiencer (경험자) | 감정/인지/지각을 경험하는 주체 | "**영희가** 슬펐다" |
| 0x2 | **THM** | Theme (대상) | 이동하거나 상태가 기술되는 대상 | "철수가 **공을** 찼다" |
| 0x3 | **PAT** | Patient (피영향자) | 행동에 의해 상태가 변하는 대상 | "**유리창이** 깨졌다" |
| 0x4 | **RCP** | Recipient (수혜자) | 무언가를 받는 대상 | "**영희에게** 책을 줬다" |
| 0x5 | **BNF** | Beneficiary (수익자) | 행동의 이익을 얻는 대상 | "**아이를 위해** 만들었다" |

### 3.2 도구/수단 (Instruments & Means)

| ID | 코드 | 역할 | 정의 | 예시 |
|----|------|------|------|------|
| 0x6 | **INS** | Instrument (도구) | 행동 수행에 사용되는 도구 | "**망치로** 못을 박았다" |
| 0x7 | **MNR** | Manner (방식) | 행동이 수행되는 방식 | "**빠르게** 달렸다" |

### 3.3 공간/이동 (Spatial)

| ID | 코드 | 역할 | 정의 | 예시 |
|----|------|------|------|------|
| 0x8 | **LOC** | Location (장소) | 사건이 발생하는 위치 | "**서울에서** 살았다" |
| 0x9 | **SRC** | Source (출발점) | 이동의 시작점 | "**집에서** 출발했다" |
| 0xA | **DST** | Destination (목적지) | 이동의 도착점 | "**학교로** 갔다" |
| 0xB | **PTH** | Path (경로) | 이동의 경유지 | "**공원을 지나** 갔다" |

### 3.4 원인/목적 (Causal)

| ID | 코드 | 역할 | 정의 | 예시 |
|----|------|------|------|------|
| 0xC | **CAU** | Cause (원인) | 사건의 원인 | "**비 때문에** 취소됐다" |
| 0xD | **PRP** | Purpose (목적) | 행동의 목적 | "**운동하러** 갔다" |

### 3.5 기타 (Others)

| ID | 코드 | 역할 | 정의 | 예시 |
|----|------|------|------|------|
| 0xE | **COM** | Comitative (동반) | 함께하는 대상 | "**친구와** 갔다" |
| 0xF | **ATR** | Attribute (속성) | 상태/속성 서술 | "하늘이 **파랗다**" |

## 4. 참여자 Edge 구조

```
PARTICIPANT Edge {
    source:     Event SIDX       // 동사 노드
    target:     Entity SIDX      // 개체 노드
    role:       4-bit            // 의미역 (0x0~0xF)
    gram_role:  2-bit (optional) // 문법적 역할 (주어/목적어/보어)
    focus:      4-bit (optional) // 강조도 (0~15 → 0.0~1.0)
    quant_ref:  TID (optional)   // 한정자 참조
}
```

### 4.1 필드 설명

| 필드 | 비트 | 설명 |
|------|------|------|
| role | 4 | 의미역 (필수) |
| gram_role | 2 | 0=미지정, 1=주어, 2=목적어, 3=보어 |
| focus | 4 | 정보적 중요도 (0=배경, 15=핵심 강조) |
| quant_ref | 16 | "모든", "대부분" 등 한정자 TID |

### 4.2 비트 레이아웃 (최소 버전)

```
[ role:4 | gram_role:2 | focus:4 | reserved:6 ] = 16-bit (1 Word)
```

## 5. 패턴 매칭 활용

### 5.1 RuleDB 패턴 예시

```
verb: acquire.v.01
participants:
  - role: AGT (0x0), type_mask: organization
  - role: THM (0x2), type_mask: organization
  - role: INS (0x6), type_mask: monetary_value (optional)
```

### 5.2 비트마스크 매칭

```
패턴: "Agent=organization, Theme=organization"

entity1.type & 0xF0 == ORGANIZATION  // 상위 4비트 체크
edge1.role == 0x0                     // Agent
```

### 5.3 능동/수동 정규화

| 표면형 | Agent | Theme |
|--------|-------|-------|
| "Apple이 Tesla를 인수했다" | Apple | Tesla |
| "Tesla가 Apple에 인수됐다" | Apple | Tesla |

MRS 파싱 단계에서 정규화 → RuleDB는 동일 패턴으로 처리

## 6. 빈도 분석

코퍼스 기반 예상 빈도:

| 순위 | 역할 | 예상 비율 |
|------|------|-----------|
| 1 | Agent | 25% |
| 2 | Theme | 30% |
| 3 | Location | 10% |
| 4 | Recipient | 8% |
| 5 | Time* | - |
| 6 | Instrument | 5% |
| 7 | 기타 | 22% |

*Time은 참여자가 아닌 동사 한정자(Period/Point)로 처리

## 7. Theme vs Patient 구분

| 역할 | 상태 변화 | 예시 |
|------|----------|------|
| Theme | 없음 (이동/기술) | "공을 **던졌다**" (공은 그대로) |
| Patient | 있음 (영향받음) | "유리를 **깼다**" (유리 상태 변화) |

실용적으로는 **Theme으로 통합**하고, 필요시 동사 의미로 구분 가능.

## 8. 예시

### 8.1 단순문

**"철수가 영희에게 책을 줬다"**

```
Event: give.v.01
├─ PARTICIPANT (AGT) → 철수
├─ PARTICIPANT (THM) → 책  
└─ PARTICIPANT (RCP) → 영희
```

### 8.2 복합문

**"비 때문에 친구와 함께 집에서 학교로 빠르게 뛰어갔다"**

```
Event: run.v.01
├─ PARTICIPANT (AGT) → [화자]
├─ PARTICIPANT (CAU) → 비
├─ PARTICIPANT (COM) → 친구
├─ PARTICIPANT (SRC) → 집
├─ PARTICIPANT (DST) → 학교
└─ PARTICIPANT (MNR) → 빠르게
```

### 8.3 상태 서술

**"하늘이 매우 파랗다"**

```
Event: be.v.01
├─ PARTICIPANT (THM) → 하늘
└─ PARTICIPANT (ATR) → 파랗다 (focus=15)
```

## 9. 동사 한정자와의 관계

| 정보 유형 | 소속 | 인코딩 위치 |
|-----------|------|-------------|
| 누가 행동했나 (Agent) | 참여자 | PARTICIPANT Edge |
| 언제 (Time) | 동사 한정자 | Tense + Period/Point |
| 누가 말했나 (Speaker) | Context/Claim | 별도 필드 |
| 확실한가 (Confidence) | 동사 한정자 | Confidence 필드 |

---

## 10. 확장 고려사항

현재 16개로 충분하나, 필요시 확장 가능:

- **Stimulus**: 경험의 자극원 ("영화가 무서웠다"의 '영화')
- **Result**: 결과물 ("케이크를 만들었다"의 '케이크')
- **Material**: 재료 ("나무로 만들었다"의 '나무')

확장 시 role 필드를 5비트로 증설 (32개)
