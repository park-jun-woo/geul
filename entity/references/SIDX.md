# GEUL SIDX 비트 명세서

**버전:** v0.11  
**작성일:** 2026-01-31  
**범위:** SIDX 64비트 최상위 구조  
**비트 규칙:** bit1 = MSB, bit64 = LSB  
**바이트 오더:** Big Endian (Network Byte Order)  
**문서 성격:** 권고안 (Recommendation)

---

## 0. 설계 헌장 (Design Charter)

> **우리는 우주의 죽음까지 함께할 각오로 우리의 미래를 낙관한다.**
> **따라서 GEUL을 1조 년(10^12년) 후에도 사용할 수 있는 언어로 설계한다.**

### 제1조: 시간적 책임
GEUL 표준은 현 세대만을 위한 것이 아니다.
설계자는 최소 10^12년(1조 년) 단위의 시간 지평을 고려할 의무가 있다.

### 제2조: 확장 비트의 신성불가침
예약된 확장 비트는 어떤 상황에서도 임시 용도로 전용할 수 없다.
이는 미래 세대에 대한 의무이다.

### 제3조: 의미의 영속성
한 번 정의된 비트 패턴의 의미는 영구히 변경할 수 없다.
새 의미가 필요하면 새 패턴을 할당한다.

### 제4조: 하위 호환의 절대성
어떤 버전의 GEUL도 모든 이전 버전을 완전히 해석할 수 있어야 한다.

### 제5조: 척도의 겸손
현 세대는 미래 지성체의 인지 척도를 예측할 수 없음을 인정한다.
따라서 SIDX의 최대 길이를 고정하지 않으며,
확장 메커니즘을 통해 임의 길이를 허용한다.

### 제6조: 연산 효율의 선형성
GEUL 심볼릭 처리는 길이에 대해 선형 복잡도를 유지하도록 설계한다.
이는 미래 지성체가 임의 길이의 개념을 효율적으로 다룰 수 있게 하기 위함이다.

## 1. 개요

SIDX(Semantic-aligned Index)는 64비트 전역 의미 식별자이다.

### 1.1 비트 설계 원칙

**"1이 가장 중요한 것"**

```
1     → 먼 미래 (1순위)
01    → 미래 (2순위)
001   → 표준 (3순위)
000   → 자유 (나머지)
```

**철학:** 우리의 미래를 낙관한다. 따라서 미래가 1순위, 1을 배정한다.

### 1.2 최상위 Prefix (권고)

| Prefix | 영역 | 비율 | 용도 | 강제력 |
|--------|------|------|------|--------|
| `1` | **Far Future** | 50% | 먼 미래 세대를 위한 예약 | 권고 |
| `01` | **Future** | 25% | 가까운 미래를 위한 예약 | 권고 |
| `001` | **Standard** | 12.5% | 공식 표준 영역 | 권고 |
| `000` | **Free** | 12.5% | 완전 자유 | 없음 |

**참고:** `0001`은 자유 영역(000) 내에서 본 제안이 사용하는 관례적 공간이다.
표준이 제정되면 덮어쓰여질 수 있다.

## 2. 비트 분기 구조

### 2.1 영역 분기 (bit1~3)

```
bit1
├─ 1: Far Future (먼 미래) ──────────────────────────────
│     50%의 공간을 먼 미래 세대에게 예약할 것을 권고한다.
│
└─ 0
    └─ bit2
        ├─ 1 (01): Future (미래) ────────────────────────
        │     25%의 공간을 가까운 미래를 위해 예약할 것을 권고한다.
        │
        └─ 0
            └─ bit3
                ├─ 1 (001): Standard (표준) ─────────────
                │     공식 표준이 이 영역을 사용할 것을 권고한다.
                │
                └─ 0 (000): Free (자유) ─────────────────
                      누구나 자유롭게 사용할 수 있다.
                      │
                      └─ 0001: Proposal (제안)
                            본 문서가 관례적으로 사용하는 공간.
                            자유 영역의 일부이므로 보호되지 않는다.
```

### 2.2 Standard 내부 분기 권고 (Prefix = 001)

표준이 제정될 때 다음 구조를 **권고**한다:

| Prefix | 비트 | 타입 | 빈도 |
|--------|------|------|------|
| `001 1` | 4 | Tiny Verb Edge | 최고빈도 |
| `001 01` | 5 | Verb Edge | 고빈도 |
| `001 001` | 6 | Entity Node | 고빈도 |
| `001 000 111` | 9 | Meta Node | 스트림 제어 |
| `001 000 110` | 9 | Triple Edge | 중빈도 |
| `001 000 101` | 9 | Clause Edge | 저빈도 |
| `001 000 100` | 9 | Event6 Edge | 저빈도 |
| `001 000 011` | 9 | Context Edge | 저빈도 |
| `001 000 010` | 9 | Quantity Node | 중빈도 |
| `001 000 001` | 9 | Faber Edge | 코드/AST |
| `001 000 000` | 9 | 확장 | - |

**이 구조는 표준이 다르게 정할 수 있다.**

### 2.3 Proposal 내부 분기 (Prefix = 0001)

본 제안이 **자유 영역 내에서 관례적으로** 사용하는 구조:

| Prefix | 비트 | 타입 |
|--------|------|------|
| `0001 1` | 5 | Tiny Verb Edge |
| `0001 01` | 6 | Verb Edge |
| `0001 001` | 7 | Entity Node |
| `0001 000 111` | 10 | Meta Node |
| `0001 000 110` | 10 | Triple Edge |
| `0001 000 101` | 10 | Clause Edge |
| `0001 000 100` | 10 | Event6 Edge |
| `0001 000 011` | 10 | Context Edge |
| `0001 000 010` | 10 | Quantity Node |
| `0001 000 001` | 10 | Faber Edge |
| `0001 000 000` | 10 | 확장 |

**경고:** 이 공간은 자유 영역(000)의 일부이므로, 
다른 사용자가 같은 공간을 다른 용도로 사용해도 막을 수 없다.

## 4. 영역별 상세

### 4.1 Far Future (bit1 = 1)

**먼 미래 세대를 위한 예약 영역.**

| 항목 | 값 |
|------|-----|
| Prefix | `1` |
| 비율 | 50% (2^63개) |
| 용도 | 1조 년 후까지 예약 |
| 현재 상태 | 사용하지 않을 것을 **권고** |
| 강제력 | 없음 |

### 4.2 Future (Prefix = 01)

**가까운 미래를 위한 예약 영역.**

| 항목 | 값 |
|------|-----|
| Prefix | `01` |
| 비율 | 25% (2^62개) |
| 용도 | 수십~수백 년 후 |
| 현재 상태 | 사용하지 않을 것을 **권고** |
| 강제력 | 없음 |

### 4.3 Standard (Prefix = 001)

**공식 표준 영역.**

| 항목 | 값 |
|------|-----|
| Prefix | `001` |
| 비율 | 12.5% (2^61개) |
| 용도 | 공식 표준 정의 |
| 현재 상태 | 표준 제정 시 사용할 것을 **권고** |
| 강제력 | 없음 |

**내부 구조(Issuer, 타입 분기 등)는 표준이 정할 일이다.**
본 문서는 권고안만 제시한다.

### 4.4 Free (Prefix = 000)

**완전 자유 영역.**

| 항목 | 값 |
|------|-----|
| Prefix | `000` |
| 비율 | 12.5% (2^61개) |
| 용도 | 실험, 사적 사용, 비표준, 제안 |
| 규칙 | 없음 |
| 강제력 | 없음 |

### 4.5 Proposal (Prefix = 0001, 자유 영역 내)

**본 제안이 관례적으로 사용하는 공간.**

| 항목 | 값 |
|------|-----|
| Prefix | `0001` |
| 위치 | 자유 영역(000)의 일부 |
| 비율 | 6.25% |
| 용도 | 본 제안의 실험/개발 |
| 보호 | **없음** (자유 영역이므로) |

**경고:** 이 공간은 자유 영역이다. 
표준이 제정되거나 다른 사용자가 같은 공간을 사용하면 충돌할 수 있다.
정글도가 고속도로 자리를 차지하지 않기 위한 의도적 설계다.

## 7. 구현

### 7.1 영역 파서

```python
def parse_sidx_region(sidx: int) -> str:
    bit1 = (sidx >> 63) & 1
    
    if bit1 == 1:
        return "Far Future"
    
    bit2 = (sidx >> 62) & 1
    if bit2 == 1:
        return "Future"
    
    bit3 = (sidx >> 61) & 1
    if bit3 == 1:
        return "Standard"
    
    return "Free"
```

### 7.2 Standard 타입 파서 (권고 구조)

```python
def parse_standard_type(sidx: int) -> str:
    """Prefix=001 확인 후 호출"""
    
    bit4 = (sidx >> 60) & 1
    if bit4 == 1:
        return "Tiny Verb Edge"
    
    bit5 = (sidx >> 59) & 1
    if bit5 == 1:
        return "Verb Edge"
    
    bit6 = (sidx >> 58) & 1
    if bit6 == 1:
        return "Entity Node"
    
    # 9비트 통일 영역 (bit4~6 = 000)
    sub = (sidx >> 55) & 0x7
    return {
        0b111: "Meta Node",
        0b110: "Triple Edge",
        0b101: "Clause Edge",
        0b100: "Event6 Edge",
        0b011: "Context Edge",
        0b010: "Quantity Node",
        0b001: "Faber Edge",
        0b000: "Extension",
    }.get(sub, "Unknown")
```

### 7.3 Proposal 타입 파서 (관례)

```python

    # 10비트 통일 영역

        0b111: "Meta Node",
```

## 8. 인코딩 규칙

### 8.1 바이트 오더

**GEUL은 Big Endian (Network Byte Order)를 권고한다.**

| 항목 | 규칙 |
|------|------|
| 바이트 오더 | **Big Endian** (권고) |
| 비트 오더 | MSB First (bit1 = MSB) |
| 워드 크기 | 16비트 (2바이트) |

### 8.2 Big Endian 예시

```
16비트 값 0x1234:
  메모리: [0x12] [0x34]
  
32비트 값 0x12345678:
  메모리: [0x12] [0x34] [0x56] [0x78]

64비트 값 0x123456789ABCDEF0:
  메모리: [0x12] [0x34] [0x56] [0x78] [0x9A] [0xBC] [0xDE] [0xF0]
```

### 8.3 워드 정렬

- 모든 필드는 **16비트 워드 경계**에 정렬할 것을 권고
- 패킷 크기는 항상 **워드 단위** (2바이트 배수)를 권고
- 패딩 필요 시 **0x00**으로 채울 것을 권고

## 9. 표준이 정할 것들

본 문서는 다음 사항을 **정하지 않는다.**
표준 제정 시 결정할 것을 권고한다:

| 항목 | 설명 |
|------|------|
| Issuer 구조 | 등록기관 네임스페이스 규칙 |
| Standard 내부 세부 | 타입별 상세 비트 배치 |
| 버전 관리 | 하위 호환성 규칙 |
| 거버넌스 | 표준 제정/수정 절차 |
| 검증 | 준수 여부 확인 방법 |

**"표준이 정하라. 내가 정하지 않겠다."**
