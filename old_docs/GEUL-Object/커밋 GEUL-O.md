# **커밋 글오 (Commit GEUL-O)**

## 1\. 정의 (Definition)

\*\*`커밋 글오(Commit GEUL-O)`\*\*는 GWMS(GEUL World Management System) 내에서 발생하는 모든 데이터 변경 사항을 기록하는 \*\*변경 이력 로그(Log)\*\*입니다. 이는 버전 관리 시스템 Git의 '커밋(Commit)' 객체에서 영감을 얻었으며, 특정 시점의 지식 상태를 보존하고 모든 변경 이력을 투명하게 추적하는 역할을 합니다.

`커밋 글오`는 데이터 자체(`문서 글오` 등)와는 명확히 분리된 메타데이터 객체입니다. 즉, **데이터의 '내용'이 아니라 데이터의 '역사'를 기록**합니다.

## 2\. 핵심 철학

  * **불변의 역사 (Immutable History)**: 일단 생성된 `커밋 글오`는 수정될 수 없습니다. 이는 과거의 특정 시점에 어떤 변경이 있었는지를 나타내는 불변의 사실 기록이기 때문입니다. 역사를 임의로 수정할 수 없는 것과 같은 원리입니다.

  * **스냅샷 기반의 버전 관리 (Snapshot-based Versioning)**: Git과 마찬가지로, `커밋 글오`는 변경된 부분(diff)만이 아닌, 변경이 일어난 시점의 \*\*데이터 전체 모습(스냅샷)\*\*을 가리킵니다. 이를 통해 특정 버전의 데이터를 손쉽게 재현하고 분석할 수 있습니다.

  * **완벽한 추적성 (Complete Traceability)**: 모든 `커밋 글오`는 자신의 부모 커밋을 가리키는 연결 체인을 형성합니다. 이 체인을 따라가면, 데이터가 최초로 생성된 시점부터 현재까지의 모든 변경 과정을 완벽하게 추적할 수 있습니다. 이는 GEUL의 '화이트박스' 철학을 시스템 수준에서 구현하는 핵심 장치입니다.

## 3\. 구조: 변경 이력의 구성요소

`커밋 글오`는 변경 사항에 대한 '누가, 언제, 왜, 무엇을'에 해당하는 정보를 다음과 같은 핵심 `글소` 패킷으로 구성합니다.

| 글소(GEUL-SO) 이름 | 핵심 역할 | 바디(Body) 내용 | 비고 |
| :--- | :--- | :--- | :--- |
| **이전 커밋 ID**\<br\>(PreviousCommitID) | 변경 이력 체인을 구성하는 연결 고리입니다. | 부모 `커밋 글오`의 **벡터 식별자**. | 최초의 커밋은 이 값이 비어 있습니다. |
| **타임스탬프**\<br\>(Timestamp) | 변경이 발생한 정확한 시점을 기록합니다. | DFC 포맷으로 표현된 시간 값. | |
| **변경 주체**\<br\>(Author) | 이 변경을 수행한 주체(사용자 또는 `글아`)를 명시합니다. | 변경 주체의 `개체 글오`를 가리키는 **벡터 식별자**. | |
| **변경 요약**\<br\>(Summary) | 변경 내용에 대한 인간이 읽을 수 있는 요약 설명입니다. | **문자열(String)**. | 예: "통계 자료 추가 및 오탈자 수정" |
| **변경 대상 루트**\<br\>(TargetRoot) | 이 커밋이 가리키는 데이터의 최상위 스냅샷(루트)입니다. | 특정 버전의 `문서 글오` 등을 가리키는 **벡터 식별자**. | 이 포인터를 통해 해당 시점의 데이터에 접근합니다. |

## 4\. `문서 글오`와의 관계

`커밋 글오`와 `문서 글오`의 관계는 Git의 `.git` 디렉토리와 실제 소스 코드 파일의 관계와 같습니다.

  * `문서 글오`는 **데이터**입니다.
  * `커밋 글오`는 **데이터에 대한 로그**입니다.
  * `커밋 글오`는 `문서 글오` 안에 **포함되지 않으며**, GWMS의 별도 로그 영역에 독립적으로 존재합니다.
  * `커밋 글오`는 `변경 대상 루트` `글소`를 통해 특정 버전의 `문서 글오`를 **참조(가리킵니다)**.

## 5\. 종합 예시: 문서 생성 및 수정 이력

1.  **초안 작성 (최초 커밋)**

    ```yaml
    ObjectID: Commit_A_ID
    GEUL_O_Type: Commit_GEUL-O

    Packets:
      - PreviousCommitID: null # 최초 커밋
      - Timestamp: "2025-09-15T10:00:00Z"
      - Author: "User_1_ID"
      - Summary: "프로젝트 제안서 초안 작성"
      - TargetRoot: "Document_Proposal_v1_ID"
    ```

2.  **내용 수정 (두 번째 커밋)**

    ```yaml
    ObjectID: Commit_B_ID
    GEUL_O_Type: Commit_GEUL-O

    Packets:
      - PreviousCommitID: "Commit_A_ID" # 부모 커밋을 가리킴
      - Timestamp: "2025-09-16T14:30:00Z"
      - Author: "User_2_ID"
      - Summary: "시장 분석 데이터 추가"
      - TargetRoot: "Document_Proposal_v2_ID" # 수정된 새 버전을 가리킴
    ```

이처럼 `Commit_B`에서 시작하여 `PreviousCommitID`를 계속 따라가면, 해당 문서의 전체 역사를 완벽하게 재구성할 수 있습니다. 이는 GWMS의 데이터 신뢰성과 감사(Audit) 기능을 위한 필수적인 기반이 됩니다.
